# 本地安装与运行指南

## 1. 环境要求

本项目第一版依赖以下本地环境：

- Node.js LTS，推荐 Node.js 20 或以上。
- pnpm。
- Python 3，推荐 Python 3.10 或 3.11。
- Python venv。
- IOPaint。
- 至少 1GB 可用磁盘空间，用于 Python 依赖、模型缓存和本地输出文件。

当前第一版不需要：

- MySQL。
- Prisma。
- Redis。
- Docker。

## 2. 前端和后端依赖

项目代码实现后，推荐使用 pnpm 管理依赖。

预计项目结构：

```text
apps/
  web/     # Vue3 前端
  server/  # Fastify 后端
storage/
  uploads/
  masks/
  outputs/
docs/
```

后续初始化项目后，常用命令预计为：

```bash
pnpm install
pnpm dev
```

具体脚本以项目 `package.json` 为准。

## 3. 安装 IOPaint

建议在项目目录中单独创建 Python 虚拟环境。

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install iopaint
```

验证 IOPaint 是否安装成功：

```bash
iopaint --help
```

如果命令不可用，确认当前终端已经激活虚拟环境：

```bash
source .venv/bin/activate
```

后端运行时不依赖终端激活虚拟环境，而是通过环境变量 `IOPAINT_BIN` 指向 IOPaint 可执行文件。默认值为：

```text
./.venv/bin/iopaint
```

## 4. 验证 IOPaint Web UI

可以先启动 IOPaint 自带 Web UI，验证本机模型是否能正常运行。

```bash
iopaint start --model=lama --device=cpu --port=8080
```

启动后访问：

```text
http://localhost:8080
```

如果 Web UI 可以正常上传图片并处理，说明本地 IOPaint 基础环境可用。

首次启动 IOPaint 时会自动下载 LaMa 模型，需要联网，耗时取决于网络情况。模型下载完成后会走本地缓存，后续启动会更快。模型缓存目录由 IOPaint/PyTorch 决定，也可以后续通过 IOPaint 的模型目录配置单独指定。

## 5. 验证 IOPaint CLI

MVP 后端会优先使用 CLI 方式调用 IOPaint。

准备一张原图和一张同尺寸 mask 后，可以手动验证：

```bash
./.venv/bin/iopaint run \
  --model=lama \
  --device=cpu \
  --image=./storage/uploads/input.png \
  --mask=./storage/masks/mask.png \
  --output=./storage/outputs/output.png
```

如果 `storage/outputs/output.png` 正常生成，说明后端后续可以通过 Node 子进程调用同样的命令。

## 6. 本地文件目录

项目运行时需要以下目录：

```text
storage/
  uploads/
  masks/
  outputs/
```

目录用途：

- `storage/uploads` 保存上传原图。
- `storage/masks` 保存前端生成的 mask。
- `storage/outputs` 保存 IOPaint 处理后的结果图。

第一版会保留这些文件，便于预览、下载和调试。

## 7. 运行策略

第一版推荐运行方式：

```text
1. 启动 Node/Fastify 后端。
2. 启动 Vue3 前端。
3. 浏览器上传图片并选择水印区域。
4. 后端通过 CLI 调用 IOPaint。
5. 前端展示处理后的图片。
```

IOPaint 不需要作为常驻 HTTP 服务运行。只有在后续发现 CLI 启动开销影响体验时，再评估切换到 IOPaint HTTP 服务模式。

### 7.1 端口与开发代理

第一版本地端口约定：

- Vue3/Vite 前端：`5173`
- Node/Fastify 后端：`3000`
- IOPaint Web UI 调试端口：`8080`

开发期前端通过 Vite `server.proxy` 将 `/api` 转发到 `http://localhost:3000`，因此后端不需要为本地开发单独开启 CORS。

结果图片由后端静态服务提供，`storage/outputs` 挂载到 `/outputs`。前端拿到 `outputUrl` 后，开发期可以直接使用后端地址访问，例如：

```text
http://localhost:3000/outputs/<uuid>.png
```

### 7.2 配置项

| 环境变量 | 默认值 | 说明 |
| --- | --- | --- |
| `PORT` | `3000` | 后端监听端口 |
| `IOPAINT_BIN` | `./.venv/bin/iopaint` | IOPaint 可执行文件路径 |
| `MAX_UPLOAD_MB` | `20` | 单文件大小上限 |
| `MAX_IMAGE_EDGE` | `4096` | 图片最长边上限 |

第一版默认本地单人使用，同一时间只处理 1 张图片。处理期间前端禁用提交按钮，后端也会拒绝新的处理请求，避免重复加载模型导致 CPU 占用过高。

## 8. 常见问题

### IOPaint 处理速度慢

第一版默认使用 CPU，处理速度受图片尺寸和机器性能影响。可以先限制上传图片尺寸，后续再评估加速方案。

### mask 和原图尺寸不一致

前端生成 mask 时必须按原图真实像素尺寸导出，不应按页面展示尺寸导出。后端也需要校验两者尺寸是否一致。

### WebP 能不能直接处理

后端会用 `sharp` 将 WebP 转换为 PNG，再传给 IOPaint。

### 为什么第一版不上数据库

当前 MVP 只需要验证本地图片去水印流程。同步接口加本地文件保存已经足够，不需要 MySQL、Prisma、任务表和历史记录。等 MVP 验证通过后，再根据实际需求决定是否引入数据库。
