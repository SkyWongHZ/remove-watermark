# 第二版功能规划与实施建议

## 1. 目标

第二版的目标是在第一版“本地单图同步去水印”已经跑通的基础上，把项目推进到更接近真实后端应用的形态。

第二版重点不是一次性做完整 SaaS，而是完成这些能力：

- 先把当前单图同步版本部署到阿里云，验证线上环境可用。
- 接入 MySQL + Prisma，让处理任务具备数据持久化能力。
- 将处理流程升级为异步任务，避免长时间 HTTP 请求挂起。
- 增加简单历史列表，让前端能看到任务状态和历史结果。
- 后端存储设计预留 OSS 扩展，但第二版不实现 OSS。

## 2. 推荐优先级

### P0：线上部署当前单图同步版本

优先级最高。

原因是当前项目最大的线上未知点是 IOPaint 能否在服务器稳定运行，包括安装、模型下载、CPU 性能、请求耗时、磁盘权限和 Nginx 上传限制。

建议先部署当前版本：

```text
Nginx
  -> Vue 静态资源
  -> Fastify API
        -> 本机 IOPaint CLI
        -> 本地 storage 文件目录
```

这一阶段不引入 MySQL、Redis、OSS 和用户系统，只验证当前功能在线上能否跑通。

验收标准：

- 域名或公网 IP 可以访问前端页面。
- 可以上传 JPG/PNG/WebP。
- 可以框选水印区域并提交。
- 后端能调用服务器上的 IOPaint。
- 处理完成后能访问 `/outputs/<uuid>.png`。
- 大图超限、IOPaint 失败、并发请求都有明确错误提示。
- 记录 1024×1024、2048×2048、4096px 三档典型图片的 IOPaint 单图耗时，用于后续决定 Nginx `proxy_read_timeout` 设置和是否必须升级异步任务。
- 部署过程中同步沉淀 `.env.example`、Nginx 配置示例、PM2 或 systemd 启动方式；如果首次部署只先跑通，也要在 P1 阶段补齐。

### P1：线上部署配置与文档

实际操作上 P1 最好与 P0 并行产出：边部署边把配置沉淀下来，避免事后只能凭记忆补文档。如果首次部署为了验证环境先快速跑通，P1 需要紧接着补齐。

建议新增：

- `.env.example`
- 线上部署文档
- Nginx 配置示例
- PM2 或 systemd 启动方式

关键配置：

```text
PORT=3000
IOPAINT_BIN=/path/to/project/.venv/bin/iopaint
MAX_UPLOAD_BYTES=15728640
MAX_IMAGE_EDGE=4096
```

Nginx 需要关注：

- `client_max_body_size`
- API 代理超时时间
- `/outputs/` 静态资源访问
- 前端静态资源缓存

### P2：MySQL + Prisma 数据持久化

MySQL + Prisma 是第二版核心能力之一，用来把项目从“图片处理工具”推进到“有后端数据层的应用”。

这里选择 MySQL，不是因为当前数据量必须用 MySQL，而是为了演练真实后端形态：独立 DB 进程、连接配置、迁移流程和任务状态持久化。

这一阶段保存任务元数据，不把图片二进制存入数据库。图片文件仍保存在服务器本地 `storage/` 目录。

推荐表：`image_jobs`

字段建议：

```text
id               String    主键，uuid
status           String    queued / running / success / failed
storageProvider  String    local / oss，第二版固定为 local
originalName     String    用户上传的原始文件名
originalPath     String    原图本地路径
maskPath         String    mask 本地路径
outputPath       String?   输出图本地路径
outputUrl        String?   前端访问结果图的 URL（local 模式下可由 outputPath 推导而冗余；保留是为了 OSS 模式下 URL 不可推导）
errorCode        String?
errorMessage     String?
width            Int?
height           Int?
fileSize         Int?
durationMs       Int?
createdAt        DateTime
updatedAt        DateTime
finishedAt       DateTime?
```

第二版默认：

```text
storageProvider = local
outputUrl = /outputs/<uuid>.png
```

第一阶段数据库只做任务记录，不做登录、权限、套餐、额度和用户隔离。

### P3：异步任务流程

接入 MySQL 后，将主流程从同步接口升级为异步任务。

异步任务不会让图片更快生成，它解决的是“HTTP 请求长时间挂起”和“前端无法查看任务状态”的问题。

推荐主流程：

```text
POST /api/jobs
  -> 接收 image + mask
  -> 保存文件
  -> 创建 queued 任务记录
  -> 返回 jobId

后台任务处理
  -> queued 改为 running
  -> 调用 IOPaint
  -> 成功写入 outputPath / outputUrl / durationMs / finishedAt
  -> 失败写入 errorCode / errorMessage / finishedAt

GET /api/jobs/:jobId
  -> 返回任务状态和结果信息
```

第二版先使用单进程 Node 后台串行处理，不引入 Redis + BullMQ。这样可以练到异步任务和状态流转，同时避免队列系统过早增加复杂度。

具体并发模型：使用 MySQL 作为任务状态来源，单进程内单 worker 按创建时间 FIFO 拉取最早的 `queued` 任务处理。新任务到达时只写入 `queued` 记录并立即返回 `jobId`，不再像 MVP 同步版本那样返回 429。

重启恢复规则：

- 服务启动时扫描 `running` 任务，并标记为 `failed`，错误码使用 `SERVER_RESTARTED`，避免永久卡在 running。
- `queued` 任务继续保留，worker 启动后继续按创建时间消费。
- 第二版不做自动重试；失败任务只展示失败原因，后续版本再考虑重试能力。

### P4：简单历史列表

简单历史列表用于体现 MySQL 和异步任务的价值。

最小功能：

- 展示最近 20 条任务。
- 展示状态：`queued`、`running`、`success`、`failed`。
- 展示原图文件名、图片尺寸、处理耗时、创建时间。
- 成功任务支持查看结果和下载结果。
- 失败任务展示失败原因。
- 当前任务可以轮询刷新状态。
- 历史列表提供手动刷新。

不做：

- 分页。
- 搜索。
- 筛选。
- 批量删除。
- 重试。
- 缩略图。

## 3. OSS 扩展预留

第二版不上 OSS，但后端设计需要为后续接入阿里云 OSS 留空间。

第二版不做：

- 不接阿里云 OSS SDK。
- 不配置 Bucket、AccessKey、RAM 权限。
- 不做签名 URL。
- 不做 CDN。
- 不做 OSS 生命周期策略。

数据库字段预留：

```text
storageProvider = local / oss
originalPath
maskPath
outputPath
outputUrl
```

后端代码结构建议预留统一存储接口：

```text
StorageService
  saveOriginal(...)
  saveMask(...)
  saveOutput(...)
  getPublicUrl(...)
```

第二版只实现：

```text
LocalStorageService
```

暂不实现：

```text
OssStorageService
```

业务逻辑应依赖统一存储接口，不要在任务处理流程里到处拼接 `storage/outputs` 和 `/outputs`。这样后续接 OSS 时，只需要新增 OSS 存储实现并调整配置，不需要重写任务、数据库和历史列表。

抽象保持最小化：一个 class、四个方法、一个文件即可，不要为单一的 `LocalStorageService` 引入 DI 容器、provider registry，也不要预先为 OSS 的签名 URL、断点续传、生命周期等留接口形参。等真要接 OSS 时再加第二个 class，到那时再确定接口细节也不迟。

## 4. 第二版接口方向

推荐第二版主接口：

```text
POST /api/jobs
GET /api/jobs
GET /api/jobs/:jobId
GET /outputs/:filename
```

接口职责：

- `POST /api/jobs`：创建任务，上传原图和 mask，立即返回 `jobId`。
- `GET /api/jobs`：返回最近 20 条任务，用于历史列表。
- `GET /api/jobs/:jobId`：返回单个任务状态，用于当前任务轮询。
- `GET /outputs/:filename`：访问本地输出结果图。

旧接口 `POST /api/remove-watermark` 在第二版直接删除，硬切到 `POST /api/jobs`。当前是单人项目，没有外部调用方，不存在过渡期成本，并行维护两套处理逻辑是纯负担。

## 5. 暂不进入第二版早期范围

以下内容不建议在第二版早期做：

- 视频去水印。
- 用户登录和权限。
- 支付、套餐、额度。
- 多租户文件隔离。
- Redis + BullMQ。
- 阿里云 OSS 实际接入。
- 批量图片处理。
- 自动识别水印。
- 自定义涂抹 mask。
- 完整任务管理页。

这些能力都有价值，但会明显增加产品复杂度。第二版应先完成线上部署、MySQL 持久化、异步任务和简单历史列表。

## 6. 建议实施顺序

推荐按下面顺序推进：

```text
1. 部署当前单图同步版本到阿里云 ECS。
2. 记录线上安装 IOPaint、Nginx、PM2/systemd 的步骤。
3. 用真实图片测试处理耗时和失败情况。
4. 接入 MySQL + Prisma，创建 image_jobs 表。
5. 抽象 LocalStorageService，并预留 storageProvider 字段。
6. 将主流程升级为异步任务接口。
7. 前端增加当前任务状态轮询。
8. 前端增加最近 20 条简单历史列表。
9. 线上回归测试任务状态、结果访问和失败记录。
```

## 7. 当前建议结论

第二版优先级最终确定为：

```text
第一优先级：线上部署当前单图同步版本
第二优先级：线上部署配置与文档
第三优先级：MySQL + Prisma 数据持久化
第四优先级：异步任务流程
第五优先级：简单历史列表
```

第二版不上 OSS，但数据库字段和后端存储接口要为 OSS 预留扩展空间。多图处理、Redis 队列、OSS 实际接入、视频去水印和自定义涂抹 mask 都放到后续版本。
